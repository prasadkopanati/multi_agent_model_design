### 2. Skill Invocations Broken in Executor Prompts

**Problem:** `prompts/build.md` and `prompts/test.md` contained lines like:
```
Invoke the agent-skills:test-driven-development skill.
```
These are Claude Code slash commands. Opencode (the executor) has no such skill system and threw:
```
skill failed
Error: Skill "test-driven-development" not found. Available skills: none
```

**Root cause:** The `STAGE_SKILLS` map in `promptCompiler.js` already specifies which skill files to inject per stage via `{{SKILLS}}`, but `build.md` and `test.md` did not use the `{{SKILLS}}` placeholder — so skills were defined but never inserted, and the prompt referenced Claude Code's internal skill system instead.

**Fix:**
- Replaced `Invoke the agent-skills:xxx skill` lines with `{{SKILLS}}` in both `build.md` and `test.md`
- Removed the `agent-skills:debugging-and-error-recovery` reference in `build.md`, replaced with plain text
- Skill content (INCREMENTAL_IMPLEMENTATION.md, TEST_DRIVEN.md, BROWSER_TESTING.md, DEBUGGING.md) is now inlined into compiled prompts at runtime

**Files changed:**
- `prompts/build.md`
- `prompts/test.md`

### 3. HITL Gates Missing After Spec and Plan Stages

**Problem:** `docs/multi-agent-guide.md` (lines 212–224) explicitly states the user must review and approve `SPEC.md` before plan runs, and approve the task breakdown before build starts. Neither gate existed — `runPipeline` in `orchestrator/orchestrator.js` iterated all stages in a plain synchronous `for` loop with no pause. The `human_required` flag in `tasks.json` was only ever set during failure escalation in `retry.js`, never checked for stage advancement.

**Fix applied to `orchestrator/orchestrator.js` only:**

1. Added `requiresApproval: true` to `spec` and `plan` pipeline entries; all other stages `false`
2. Added `APPROVAL_ARTIFACTS` map pointing to `SPEC.md` and `tasks/plan.md`
3. Added `promptApproval(stage, workspace)` — uses Node built-in `readline`, prints artifact path, asks `[y/N]`
4. Made `runPipeline` async; approval gate fires after each `requiresApproval` stage
5. Rejection (`n` or Enter) exits cleanly with message; approval (`y`) continues
6. Entry-point call updated to `.catch()` the Promise

**Key design decision:** Approval gates are **unconditional** — YOLO mode does NOT bypass spec/plan gates. YOLO is appropriate only for automated executor stages (build, test, review). This was explicitly confirmed by the user.

**File changed:**
- `orchestrator/orchestrator.js`
